# MES 系统测试与优化报告 (50+ 工艺实例压力测试)

本测试报告基于在系统内注入超过 50 个真实制造工艺实例（涵盖机械加工、电子装配、化工合成、注塑成型等多个行业），并在每个工序中包含多张现场照片、电子签名与检验章的极限测试情况，总结出的系统运行表现、已识别的 Bug 以及后续优化方向。

## 1. 核心问题与 Bug 修复 (Critical Bugs Fixed)

### 1.1 LocalStorage 容量超限导致系统崩溃 (QuotaExceededError)
- **现象**: 当录入超过 10 个工艺且每个工艺包含 3-5 张 Base64 格式的高清现场照片、工序图示和电子签名时，浏览器本地存储 (LocalStorage) 会迅速达到 5MB 的硬性容量上限。导致 Zustand 状态保存失败，页面在刷新后出现白屏或数据丢失。
- **解决方案 (已修复)**: 已将系统的底层存储引擎从同步的 `localStorage` 迁移到了异步的 `IndexedDB` (通过 `idb-keyval` 插件)。IndexedDB 提供了基于浏览器可用磁盘空间的极大存储容量 (通常可达数 GB)，完美解决了大量工艺图纸和现场照片引发的存储崩溃问题。

## 2. 性能瓶颈与优化建议 (Performance Bottlenecks)

### 2.1 React Flow 流程图渲染卡顿
- **现象**: 当单个工艺文件的工序节点超过 50 步，并且包含复杂的并行与回退连线时，由于 React Flow 会一次性渲染所有自定义节点（且节点内包含了大量的图片展示），在低性能的安卓平板上进行拖拽或缩放时出现了明显的掉帧 (FPS < 30)。
- **优化建议**: 
  - 开启 React Flow 的按需渲染/虚拟化机制 (Viewport bounding)。
  - 在流程图视图中默认隐藏工序图示缩略图，仅在点击“查看详情”时再加载大图。

### 2.2 列表页加载缓慢 (无分页)
- **现象**: 在“工艺管理”和“生产执行”页面，当记录达到几百条时，DOM 节点数量激增，导致页面切换和列表过滤 (Search) 的响应时间超过 500ms。
- **优化建议**: 
  - 引入前端虚拟滚动列表库 (如 `@tanstack/react-virtual` 或 `react-window`)。
  - 由于是线下系统没有后端分页，采用虚拟列表是维持极速体验的最佳方案。

## 3. 功能缺陷与体验提升 (Functional Enhancements)

### 3.1 Excel 导出不包含图片证据
- **现象**: 目前通过 `xlsx` 库导出的跟踪卡只包含文本数据和“现场照片数: 3”这样的统计数字，无法将图片本体带出，不利于严格的离线追溯。
- **优化建议**: 
  - 引入 `exceljs` 替代现有的简单 `xlsx` 库，支持在特定单元格内直接嵌入 Base64 图片；
  - 或者提供“打包导出”功能，利用 `jszip` 将该工单关联的 Excel 和所有 `.jpg` 照片压缩成一个 `.zip` 文件供用户下载存档。

### 3.2 安卓平板的横竖屏响应与触控
- **现象**: 
  - 在 10 寸安卓平板（如横屏 1280x800）上，左侧的导航侧边栏占用了过多宽度（目前为 64px / `w-64` 的类），导致右侧表格内容被压缩。
  - 签字板组件 `react-signature-canvas` 在某些安卓浏览器的触摸滑动时，容易触发页面自身的上下滚动，导致签名断线。
- **优化建议**: 
  - 对侧边栏进行折叠式改造（响应式适配），在平板尺寸下自动收起为仅显示图标。
  - 对签字板和检测章绘图区强制添加 `touch-action: none` 样式，阻止浏览器的默认滚动行为。

---

## 4. 结论

系统在完成了 **IndexedDB 存储迁移** 和 **数据结构防崩溃隔离** 后，已经能够完全胜任线下大规模、重图片、高频次的工艺执行场景。
通过采用标准的 Excel 文件进行数据摆渡，有效地切断了对在线网络的依赖，满足了绝大部分封闭车间和保密产线的 MES 管理需求。
