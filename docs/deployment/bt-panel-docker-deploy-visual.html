DATC-PMS / 宝塔部署图文版

当前项目在服务器上进行 Docker 部署的详细步骤（/pms 子路径版）

本指南适用于当前项目的实际结构：Vite 前端单页应用 + Docker 多阶段构建 + Nginx 容器托管 + 宝塔面板反向代理 + /pms 子路径访问。 你可以把这份 HTML 直接发给运维或部署人员，照着一步步操作即可。

1

准备服务器与宝塔环境

先把基础环境准备好，否则后面的命令无法执行。

这一步的作用：确保服务器具备运行 Docker 容器、创建站点、配置反向代理和 HTTPS 的基础能力。
服务器系统：Ubuntu / CentOS 宝塔面板已安装 推荐 2 核 2G
  • 进入宝塔面板，安装 Docker 管理器。
  • 安装 Nginx，因为宝塔后续要做网站和反向代理。
  • 安装 Git，方便直接从 GitHub 拉代码。
  • 云服务器安全组或防火墙放行 `80`、`443`，调试阶段可临时放行 `8080`。
2

确认项目里的部署文件

当前项目已经补齐 Docker 部署所需文件，不需要你手工再写。

这一步的作用：让部署过程标准化、可复用，不再依赖服务器手工安装 Node 环境和手动复制构建产物。

`Dockerfile` 的作用

先在 Node 环境里执行 `npm run build`，再把生成的 `dist` 放进 Nginx 容器中的 `/pms/` 目录提供服务。

`nginx.conf` 的作用

让前端静态资源在 `/pms/` 下正常访问，并通过 `try_files` 支持单页应用刷新不 404。

`docker-compose.yml` 的作用

固定容器名、端口和自动重启策略，后续更新时只用一条命令重建。

`.dockerignore` 的作用

排除 `node_modules`、`.git` 等无关文件，减少构建体积并提升速度。

3

从 GitHub 拉取当前项目

在服务器上把最新代码拉下来，后面的容器构建会直接基于这份代码执行。

这一步的作用：把本地已经同步到 GitHub 的 Docker 文件、Nginx 配置和前端代码完整拉到服务器。
cd /www/wwwroot git clone https://github.com/Simon201503/DATC-MES-SourceCode.git cd DATC-MES-SourceCode

如果服务器上已经有项目目录，改用下面的更新命令：

cd /www/wwwroot/DATC-MES-SourceCode git pull origin main
4

用 Docker Compose 构建并启动容器

这是最核心的一步，会自动完成构建前端、生成镜像、启动 Nginx 容器。

这一步的作用：把前端项目从源代码变成可以在服务器里长期运行的容器服务。
cd /www/wwwroot/DATC-MES-SourceCode docker compose up -d --build
  • `--build`：强制按最新代码重新构建镜像。
  • `-d`：让容器在后台运行，不占用当前终端。
  • 当前配置会把服务器 `8080` 端口映射到容器 `80` 端口。
  • 系统最终访问入口是 `/pms/`，根路径只显示占位页。
docker compose ps docker logs datc-pms-web

先用以下地址测试容器是否跑起来：

http://服务器公网IP:8080/pms/
5

在宝塔中添加网站并做反向代理

让用户不需要带 `:8080` 端口访问，而是直接通过正式域名打开 `/pms/` 系统。

这一步的作用：把“容器内部服务”转成“标准网站访问入口”，再由容器内部 Nginx 继续处理 `/pms/` 子路径。
  • 打开宝塔面板，进入 网站。
  • 点击 添加站点，填写你的域名，例如 `pms.example.com`。
  • PHP 版本选择静态即可，这里不需要 PHP 运行环境。
  • 进入站点设置，打开 反向代理。
  • 新增反向代理，目标地址填写：
http://127.0.0.1:8080

这里使用 `127.0.0.1` 的原因是：宝塔和容器都在同一台服务器上，本机转发更稳定，也不需要对外暴露容器端口。

6

配置域名解析与 HTTPS

让访问地址变成正式可分享的 HTTPS 域名。

这一步的作用：解决“公网怎么访问”和“浏览器提示不安全”的问题。
  • 到域名服务商后台，添加 `A` 记录，把域名解析到服务器公网 IP。
  • 回到宝塔，打开站点设置中的 SSL。
  • 选择 `Let's Encrypt` 自动申请证书。
  • 签发成功后开启 强制 HTTPS。
完成效果：用户最终访问的是 `https://你的域名/pms/`，而不是 `http://IP:8080/pms/`。
7

为什么这个项目在 /pms 下刷新子路由不会 404

因为项目是单页应用，子路由例如 `/pms/process/list` 需要专门处理。

这一步的作用：解释为什么必须保留当前 `nginx.conf` 中 `/pms/` 的 SPA 回退配置。
location = /pms { return 301 /pms/; } location /pms/ { try_files $uri $uri/ /pms/index.html; }
  • 当用户访问 `/pms/process/list` 时，Nginx 先尝试找真实文件。
  • 如果服务器上没有这个物理路径，就回退到 `/pms/index.html`。
  • 再由 React Router 在前端接管路由显示正确页面。

以后项目更新时怎么做

以后本地改完代码并推送到 GitHub 后，服务器只需要执行更新命令。

这一步的作用：用固定流程完成版本更新，不必重新手工部署整个站点。
cd /www/wwwroot/DATC-MES-SourceCode git pull origin main docker compose up -d --build
  • `git pull origin main`：拉取最新代码。
  • `docker compose up -d --build`：按最新代码重新构建镜像并重启服务。
9

常见问题排查

部署时最常见的问题通常集中在容器、端口、域名和反向代理四类。

页面打不开

先执行 `docker compose ps` 看容器有没有正常运行。

容器启动失败

执行 `docker logs datc-pms-web` 查看构建或启动日志。

域名访问异常

检查域名解析、宝塔反向代理地址、Nginx 服务状态。

刷新页面 404

确认 `nginx.conf` 中保留了 `try_files $uri $uri/ /pms/index.html;`。

端口冲突

如果 `8080` 被占用，可把 Compose 改为其他端口，例如 `8090:80`。

HTTPS 不生效

确认域名已经解析到服务器，并且宝塔证书签发成功。
